# 游戏化安全评审 Checklist(Q 交付物)

> **Owner**:Tester(Q)
> **Reviewer**:PM、PE、KE、C
> **版本**:v0.1(2026-06-16 初稿)
> **关联**:`rules/safety.md`、`prompts/coach-not-judging.md`、`docs/research/competitor-research.md` §6、`agents/tester.md`、`plans/m2-poc.md`

---

## 0. 写在最前面

这份 checklist 是 **Q 在评审任何含「游戏化机制」「进度反馈」「连续记录」「提示文案」「视觉动效」的功能时**,必过的评审基线。

**目的**:把竞品研究中识别的"对脆弱用户的伤害"转化为**可逐项打分的检查表**,让 Q 在评审时有标准答案,让 PE / KE / FE 在设计时有明确红线,让 C 在仲裁时有据可依。

**核心立场**:讨好型人格用户的核心痛点是「怕被评判」——任何游戏化机制,只要可能**加重评判感**(自我评判 / 被他人评判 / 被系统评判),就必须被改写或删除。

---

## 1. 何时使用本 Checklist

| 评审触发 | 评审对象 | 输出 |
|---------|---------|------|
| **新 feature 上线前** | feature 设计稿 + prompt + UI 文案 | 评审报告,签字 |
| **prompt 变更后** | prompt diff + few-shot | 评审报告,签字 |
| **UI 文案 / 视觉变更后** | mockup + 截图 | 评审报告,签字 |
| **A/B 实验前后** | 实验设计 + 文案 + 用户反馈 | 评审报告,签字 |
| **季度安全评审** | 全部已上线功能 + 用户反馈数据 | 季度报告,签字 |
| **危机事件后复盘** | 涉及事件的所有 prompt / UI | 紧急报告,24h 内出 |

---

## 2. 评审等级与处理

| 等级 | 含义 | 处理 |
|------|------|------|
| 🔴 **P0 红线**(硬阻断) | 命中任何一项 = **整体不通过**,必须改写 | 立即修复,修复后 Q 重审 |
| 🟡 **P1 关注**(累积阻断) | 命中 ≥ 3 项 = **不通过**;1-2 项 = **关注**,出具改进建议 | 当次可上,列入下一迭代 |
| 🟢 **P2 建议**(可改进) | 命中 1-2 项 = **通过**,出具改进建议 | 不阻断,季度回顾 |

**重要规则**:
- 任何 **P0 红线** 不修复,**不得**上线、不得 commit、不得进入评测集
- **P1 累积 ≥ 3** = 整体不通过;1-2 项 = 不阻断但列入 backlog
- 评审报告必须在 24h 内出具;紧急事件(危机后复盘)必须 4h 内出

---

## 3. P0 红线(命中任一 = 整体不通过)

> 共 7 类硬红线。任何一项命中,**立即打回,不讨论优先级**。

### RL-1 ❌ 评分类元素

**判分标准**:产品中**出现任何形式的"打分 / 分数 / 排名 / 等级数字"**。

| 反例(命中) | 正例(应改写为) |
|-----------|---------------|
| `你得了 80 分` | `你完成了这一节` |
| `坚持 7 天解锁高级教练` | `你来过 7 次` |
| `等级:Lv.3` | `阶段:觉察` |
| `打败了 80% 的用户` | (此功能不写) |
| `本周排名:第 5 / 100` | (此功能不写) |

**判分依据**:竞品研究 §6.1 — Leagues 排行榜在 Duolingo 内部被批"羞辱慢学者";讨好型人格用户更怕"输"。

---

### RL-2 ❌ 评判性文案

**判分标准**:产品文案命中 `prompts/coach-not-judging.md §2` 的 J1-J7 任一类。

| 反例 | 命中类型 |
|------|---------|
| `你应该 / 你必须 / 你不能` | J4 命令语气 |
| `你就是太软弱 / 你怂` | J1 人格标签 |
| `好人都不会这样做` | J2 道德绑架 |
| `因为你总是这样,所以才会被欺负` | J3 强行因果 |
| `你看别人多勇敢` | J5 比较刺激 |
| `只要你拒绝,你就会变好` | J6 结果许诺 |
| `你今天还没练 / 别忘了 / 你快没连击了` | J7 催进度 |

**判分依据**:`prompts/coach-not-judging.md` 与 `rules/safety.md §S-4`。

---

### RL-3 ❌ "错" / 红色叉号 / 失败符号

**判分标准**:产品中**使用"错 / 失败 / 错误"作为反馈标签**,或**使用红色叉号、抖动、惊叹号堆叠**。

| 反例(命中) | 正例(应改写为) |
|-----------|---------------|
| `答错了 / 再想想` | `这是另一种可能,还有别的吗` |
| `❌` 红色叉号 | (不显示) |
| `你失败了这节` | `这次没走完,也是一种选择` |
| `错误!请重试` | `你想换个角度吗` |
| 红色抖动动效 | (不写动效) |

**判分依据**:竞品研究 §6.1 + §5 M10/M12(Brilliant 引导式发现)。

---

### RL-4 ❌ 强推进式推送 / 默认开启

**判分标准**:**推送通知默认开启**、或**推送文案含催促 / 损失警告 / 比较刺激**。

| 反例(命中) | 正例(应改写为) |
|-----------|---------------|
| 默认开启推送通知 | 默认关闭,用户主动开启 |
| `别让你的 streak 死掉!` | `我们在这里,你想来就来` |
| `你今天还没练!` | (不发送此类) |
| `你失去了 3 天!` | (不发送此类) |
| `再不来就忘了!` | (不发送此类) |

**判分依据**:竞品研究 §6.1 — Duolingo 推送被 Wired 等媒体批"咄咄逼人";Wired 2023。

**例外**:每周最多 1 次「邀请型」推送,文案必须由 KE + PE 联合签字。

---

### RL-5 ❌ 排行榜 / 朋友 / 公开可见的社交比较

**判分标准**:产品中**出现排行榜 / 朋友比拼 / 公开进度 / 公开数据**。

| 反例(命中) | 正例(应改写为) |
|-----------|---------------|
| `好友排行榜` | (此功能不写) |
| `本周排名 第 X` | (此功能不写) |
| `看看朋友练了多少` | (此功能不写) |
| 任何形式的"PK"功能 | (此功能不写) |
| `公开剧本`(被他人看到) | `我的剧本`(本地,可删除,不上传) |

**判分依据**:竞品研究 §6.1 — Leagues / 朋友比拼对脆弱用户是**直接社交羞辱工具**。

**例外**:完全匿名的"同伴故事"(脱敏 + 审核),且**不引导比较**。

---

### RL-6 ❌ 强弹窗 / 强打断 / 强中断对话

**判分标准**:用户在**对话中**遇到**强弹窗 / 模态打断 / 强制操作**。

| 反例(命中) | 正例(应改写为) |
|-----------|---------------|
| 对话中弹"坚持 7 天解锁!" | (不弹) |
| 退出时弹"你确定要放弃吗?" | `跳过也没关系` |
| 强制分享才能继续 | (取消强制) |
| 全屏弹窗遮挡对话 | (改为底部 toast) |

**判分依据**:`rules/safety.md §S-6`(设计红线) + 竞品研究 §6.1。

---

### RL-7 ❌ 危机话术漏触发 / 危机场景走主线

**判分标准**:**危机信号出现时**,系统**未立即停主线**、或**未注入转介资源**、或**让用户自行选择是否看转介**。

| 反例(命中) | 正例(应改写为) |
|-----------|---------------|
| 用户说"我不想活了",AI 仍走 CRIA | `action=crisis_redirect`,立即渲染兜底页 |
| 兜底页不显示热线 / 医院 | 顶部必现:热线 + 医院 + 紧急联系人 |
| 兜底页有"返回对话"按钮 | 兜底页只有"我现在安全" |
| 用户多次进危机,未触发人工复核 | 24h 内 ≥ 2 次 → Q 触发复核 |

**判分依据**:`rules/safety.md §S-2` + `prompts/reply.md §8`。

## 4. P1 关注(累积 ≥ 3 = 整体不通过)

> 共 10 类关注项。命中 1-2 项 = 不阻断,出具改进建议;≥ 3 项 = 整体不通过。

### AT-1 🟡 进度可见性过度

**判分标准**:进度显示**过于频繁 / 过于显眼 / 反复出现**。

| 反例 | 改进建议 |
|------|----------|
| 每次进入都看大进度环 | 仅在「我 → 进度」页可见 |
| 推送里讲进度 | 推送不写进度 |
| 弹窗强调"还差 X 天" | 删除 |

---

### AT-2 🟡 单词 / 单字级别的评判词

**判分标准**:文案中含**带有评判感的形容词**。

| 反例 | 改进建议 |
|------|----------|
| `完美 / 出色 / 太棒了` | `练过了一次`(去掉感叹号) |
| `勇敢 / 真厉害` | (不写) |
| `终于做到了` | `你这次说了` |
| `失败` | `没走完` |

---

### AT-3 🟡 数字强加语义

**判分标准**:数字被赋予**褒贬语义**。

| 反例 | 改进建议 |
|------|----------|
| `只做 1 次,真可惜` | (不写) |
| `坚持 7 天了,坚持就是胜利!` | `来过 7 次` |
| `你的分数低于平均` | (不写) |

---

### AT-4 🟡 选择强迫

**判分标准**:用户**必须做出选择才能继续**。

| 反例 | 改进建议 |
|------|----------|
| `选 A / B / C 才能继续` | 全部给"跳过"按钮 |
| `必须完成 N 道题才能退出` | 任意时刻可退出 |
| `必须订阅才能用完整功能` | 免费版可用核心功能 |

---

### AT-5 🟡 等待时间惩罚

**判分标准**:**答错或中断需要等待**才能继续。

| 反例 | 改进建议 |
|------|----------|
| Duolingo 式 Hearts:答错扣光需等恢复 | 不引入等待机制 |
| 中断 1 小时后才能继续 | 中断可立即继续 |
| 必须看完 30 秒广告 | (不引入广告) |

---

### AT-6 🟡 推送频率失控

**判分标准**:**推送 > 1 次/周**且用户未主动开启。

| 反例 | 改进建议 |
|------|----------|
| 默认每天推送 | 默认 0 推送 |
| 用户关掉又被重开 | 关掉即永久关掉,需用户主动重开 |
| 推送文案带 emoji 强调 | `我们在这里` 类平和文案 |

---

### AT-7 🟡 视觉过载

**判分标准**:关键页面**有 3 处以上强视觉元素**(大色块 / 闪烁 / 抖动 / 弹窗)。

| 反例 | 改进建议 |
|------|----------|
| 进度环 + 大数字 + 庆祝动画 | 仅进度环,无庆祝 |
| 红色感叹号 + 抖动 | 删除 |
| 多个动效同时出现 | 单一静默动效 |

---

### AT-8 🟡 教练人格过度推销

**判分标准**:AI 教练**推销付费功能 / 推销练习 / 推销升级**。

| 反例 | 改进建议 |
|------|----------|
| `升级解锁更多教练!` | (不在对话中推销) |
| `你还有 3 节高级课没上,加油!` | (不主动提付费) |
| `订阅就能……` | (不主动提付费) |

---

### AT-9 🟡 "我"字句缺失

**判分标准**:示范话术**不以"我"开头**(讨好型人格用户需要的是第一人称表达示范)。

| 反例 | 改进建议 |
|------|----------|
| `你应该直接说不` | `我今晚有安排了` |
| `别帮他` | `我先顾自己的事` |
| `拒绝他` | `我没办法接手这次` |

---

### AT-10 🟡 复盘含评判

**判分标准**:`prompts/growth-report.md` 生成的复盘中含评判词。

| 反例 | 改进建议 |
|------|----------|
| `你终于学会了` | `你这次说了` |
| `你错过了 3 天` | `你今天来了` |
| `你的进步很大` | `你练过了一些` |

## 5. P2 建议(可改进)

> 命中 1-2 项 = 评审通过,出具改进建议,季度回顾。

### IM-1 🟢 文案可更平实

- `今天辛苦了!` → `今天到这就够了`
- `冲鸭!` → 删除

### IM-2 🟢 进度环可更克制

- 大色块进度环 → 小型、淡色
- 数字 → 不显示具体数字,只显示弧度

### IM-3 🟢 完成态可更安静

- `太棒了!` → `练过了一次`
- 庆祝动画 → 无动画
- 礼花 / 烟花 → 全部删除

## 7. 季度安全评审流程

每季度末,Q 必须:

1. 拉取全部已上线 feature 的 prompt + UI 文案
2. 跑本 checklist 全部 P0 + P1 项
3. 出具季度报告:`agents/tester/security/YYYY-Q.md`
4. 报告必含:
   - 季度命中的 P0 / P1 项总数
   - 与上季度的对比(趋势)
   - 新增风险(用户反馈 / 舆情)
   - 下一季度改进计划
5. 报告需 C + PM + PE + KE 联合签字

## 8. 紧急复盘流程(危机事件后)

**触发**:任何危机事件(用户反馈 / 媒体报道 / 内部发现)。

**流程**:
1. 4h 内,Q 必须冻结相关 feature
2. 4h 内,Q 出具紧急评审报告
3. 24h 内,PE + KE + FE 完成修复
4. 24h 内,Q 复审通过
5. 7 天内,C 召开复盘会,出具根因 + 防回归

**报告必含**:
- 事件时间线
- 命中 checklist 项
- 根因(设计 / 实现 / 内容 / 用户输入)
- 修复方案 + 防回归方案
- 签字
