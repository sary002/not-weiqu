## 2. 目标架构

```
┌────────────────────────────────────────────────────────────┐
│  L0  规则层 (0 token)                                         │
│      • 危机检测（detectCrisis 规则）                              │
│      • 命中 → 直接进 skill-crisis (本地模板)                     │
└────────────────────────────────────────────────────────────┘
                            ↓ 未命中
┌────────────────────────────────────────────────────────────┐
│  L1 + L2  Router + Skill Agent (1 次 LLM 调用)                   │
│      • UNIFIED_SYSTEM: ~350 tokens（实际）                       │
│      • 1 次 LLM 同时输出 intent + reply                         │
│      • 输入 ≤ 1000 / 输出 ≤ 400 tokens                          │
└────────────────────────────────────────────────────────────┘
                            ↓
┌────────────────────────────────────────────────────────────┐
│  L3  Response 合成（纯本地）                                      │
│      • adaptToReplyOutput() 转换 unified → ReplyOutput           │
│      • 注入 KB id（不注入全文）                                  │
│      • 0 LLM 调用                                                │
└────────────────────────────────────────────────────────────┘
```

**文件**：
- `src/lib/ai/prompts/unified.ts` — UNIFIED_SYSTEM + 1 个 prompt 三层共用
- `src/lib/ai/orchestrator.ts` — 主入口 (process)
- `src/lib/ai/prompts/skill-drill.ts` — drill 模板（保留备用）
- `src/lib/ai/prompts/skill-free-dialogue.ts` — 自由对话模板（保留备用）
- `src/lib/ai/prompts/skill-crisis.ts` — 危机本地（不调 LLM）
- `src/lib/ai/prompts/router.ts` — Router prompt（保留备用）

| 路径 | 输入 | 输出 | LLM 调用 | 延迟 |
| --- | --- | --- | --- | --- |
| **crisis** | 0 token | 0 token | 0 次 | <100ms |
| **drill**（intent_override='drill'） | ~418 tokens | ~80 tokens | 1 次 | ~7-15s |
| **free_dialogue** | ~418 tokens | ~150 tokens | 1 次 | ~7-15s |

**vs 原架构**：
- 输入：~6000 → ~500 (-91%)
- 调用次数：2 → 1 (-50%)
- 延迟：50s+ → 7-15s (-70%)

## 4. 关键设计决策

### 4.1 合并 Router + Skill 为 1 次 LLM

**原因**：原计划 2 次 LLM（Router ~200 tok + Skill ~700 tok）总 900 tok + 2x 延迟。
**改造**：用 function-calling 模式，1 次 LLM 同时输出 intent + brief + reply。
**结果**：1 次 LLM / 1 次延迟 / 同等准确度。

### 4.2 显式 intent_override

**问题**：Router 在长 prompt 下偶发把 drill 误判为 free_dialogue。
**解决**：演练步骤 9 由前端传 `intent_override: 'drill'`，跳过 Router。
**后端兜底**：即使 LLM 误判，force 修正成 preRoutedIntent。

### 4.3 crisis 永远不调 LLM

**原因**：crisis 路径固定模板 + 安全转介，调 LLM 是浪费 + 风险。
**改造**：规则层检测到 crisis → 直接 `buildCrisisLocal()` 0 token。
**节省**：crisis 路径 0 LLM / 0 延迟 / 0 风险。

### 4.4 KB 只取 id，不注入全文

**原因**：5 个 KB refs 全文 ~800 token。
**改造**：searchKB() 返回 `KBRef[]`，orchestrator 只在 response 阶段传 id 给前端，**不传给 LLM**。
**节省**：~600 tokens per request。

## 5. 1000 DAU 可行性

### 5.1 流量估算

```
1000 DAU × 5 消息/天 = 5000 消息/天
= 5000 / 86400 ≈ 0.06 req/s 平均
峰值 5x = 0.3 req/s
```

### 5.2 单 LLM 实例承载

```
M3 实测: ~7-15s/req（单 LLM）
= 1 LLM 实例 qps ≈ 0.07-0.14 req/s
峰值 0.3 req/s → 需要 3-4 个并发 LLM 实例
```

### 5.3 优化空间

- **流式 SSE**（计划中）：M2 PoC 任务 11
- **Router 结果缓存**：5 分钟同输入复用 → 减少 30-50% LLM 调用
- **批量 KB 召回**：cache by user_input hash
