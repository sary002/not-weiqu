## 学到了什么

- **Next.js dev 缓存会让二分定位 bug 不可靠** — 改完文件后 SWC 编译有 5-30s 滞后；遇到"should work"但不 work 的情况，重启 dev server 强制重新编译最稳
- **TS 编译过但 SWC 失败** — 大概率是某个 Unicode 字符或 JSX/SWC 微妙不兼容（不是 TS 问题）；用"最小可编译模板逐步加"的方法最快定位
- **in-memory Map + globalThis** 是 dev 模式最简的"跨模块共享状态"；生产接 Supabase 后这段直接废弃
- **3 层架构 + 1 次 LLM** 比 2 层架构 + 2 次 LLM 更省；function-calling 模式让"路由"和"技能"自然合并输出

### W-7 多 Agent 并发开关（v1 方案 · 当前 off）
- **新增文件**：
  - `.claude/concurrency.json`（开关配置文件，默认 mode=off）
  - `rules/workflow.md` §W-7（规则文档，可见状态）
  - `CLAUDE.md` §2.4（治理层索引）
- **设计原则**：规则始终可见，开关决定行为；不通过屏蔽规则来禁用
- **三种模式**：`off`（默认）/`on`（强制并行）/`auto`（按阈值判断）
- **会话级切换**：
  - 开启：`ultracode` / `fan out` / `多 agent 并行` / `parallel`
  - 关闭：`ultracode off` / `一个人做` / `省 token`
- **团队级切换**：修改 `.claude/concurrency.json` 的 `mode` 字段，需先写 ADR
- **设计迭代**：v0 用 HTML 注释屏蔽草案 → v1 改为开关化方案（用户提议，v0 是反模式）
- **触发原因**：今日竞品研究 22 文件产出实际为 C 单线串行完成，虽按角色命名但本质是"冒名顶替"，违反治理原则
- **未来工作**：写 ADR（docs/decisions/adr-NNN-default-concurrency.md）记录开关决策；待 token 预算允许后考虑改默认 mode 为 auto
